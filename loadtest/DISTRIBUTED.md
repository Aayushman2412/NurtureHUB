# Distributed load test — validating the 7,000-user exam

One machine cannot open 7,000 real connections, so the load is generated by a
Locust **master** + many **workers** spread across cores/boxes, all pointed at
your **staging** backend (never production). This runbook walks the exact steps
to answer "does our server handle 7,000 simultaneous exam-takers?".

> ⚠️ Run this against staging on the **production-grade hardware**, seeded with
> test accounts — and tell your host/provider first. A 7k flood looks like a DoS.

## 0. Prerequisites

- The hardened backend deployed on staging (the `perf/loadtest-hardening` work,
  now on `main`), ideally via the scale overlay: pgbouncer + multiple workers
  (see `../DEPLOY.md`). Note how many app workers and what Postgres
  `max_connections` you configured — that's what you're validating.
- Docker on the load-generator box(es).
- Enough generator CPU: budget ~1 vCPU per ~500–800 virtual users, so 7k ≈
  10–14 worker vCPUs. One 16-core box can do it; otherwise spread workers across
  boxes.

## 1. Seed test accounts on staging

The generators log in (or pre-mint tokens) as `loadtest_1..N@nhload.org`. Seed at
least as many as the peak users, with headroom:

```bash
# on the staging DB host (points at staging's Postgres)
NH_LOADTEST_DB=postgresql://USER:PASS@STAGING_DB/DBNAME \
  python seed_accounts.py --count 8000 --complete-tutorials
# --complete-tutorials unlocks the exam so users can start it immediately
```

Activate the exam once (admin API), so all users can enter it:

```bash
NH_HOST=https://staging.example.org python setup_env.py \
  --activate 5 6 --questions 20 --district meghalaya
```

## 2. Build the generator image

```bash
cd loadtest
docker compose -f docker-compose.loadgen.yml build
```

## 3. Run the staged climb (headless): 1k → 3k → 5k → 7k

This walks the load up in stages and stops, so you see *at what user count* the
first failures appear. `ExamHerdUser` is the exam profile (start → proctoring
WebSocket → synchronized submit).

```bash
NH_HOST=https://staging.example.org \
NH_SHAPE=1 \
NH_SHAPE_STAGES="1000:200:180, 3000:300:180, 5000:400:180, 7000:500:300" \
LOCUST_CLASS=ExamHerdUser \
MASTER_EXTRA_ARGS="--headless --csv /loadtest/run --expect-workers 8" \
  docker compose -f docker-compose.loadgen.yml up --scale worker=8
```

- Each `NH_SHAPE_STAGES` entry is `users:spawn_rate:hold_seconds`.
- `NH_PREMINT_TOKENS=true` (default) mints JWTs so the run measures exam capacity,
  not the login funnel — set `NH_JWT_SECRET` to staging's real secret. To also
  test the login storm, set `NH_PREMINT_TOKENS=false` (and expect the per-IP rate
  limit unless you raise `RATE_LIMIT_LOGIN` / set `TRUST_PROXY_HEADERS`).
- Watch the target: the backend's own logs, `pg_stat_activity`, and pgbouncer
  stats. The first thing to saturate (WebSocket connect, DB connections, CPU) is
  your real bottleneck.

## 4. Or drive it live from the web UI

Drop `NH_SHAPE` and `--headless`; open `http://<master>:8089` and set users +
spawn rate by hand. Good for interactive probing before a scripted run.

## 5. Reading the result

- **`run_stats.csv` / the master summary** — failure rate + p50/p95/p99 per stage.
  Capacity = the highest stage that stayed at ~0 failures with acceptable p95.
- **WebSocket connect failures first** → app-worker / event-loop bound: add workers.
- **`QueuePool limit reached` / `too many clients`** → DB connection bound: raise
  pgbouncer `default_pool_size` + Postgres `max_connections`, shrink per-worker
  `DB_POOL_SIZE`.
- **Slow submits under the end-of-exam burst** → DB write bound: faster disk,
  batch, or shard.

## Caveats

- **The herd barrier is per generator process.** Each worker's cohort submits
  together, which produces a heavy, realistic submit burst — but it is not one
  globally-synchronized millisecond-aligned submit across the whole fleet. For a
  hard global spike, back the barrier with Redis (a small change to `locustfile.py`).
- **Live admin monitoring at scale needs Redis** on the *backend* side: the
  WebSocket connection manager is per-process, so an admin only sees candidates on
  its own worker until Redis pub/sub is added. Candidate sockets themselves are
  fine across workers. Factor this in if proctors watch the live exam.
- Generator saturation masquerades as target slowness — keep each generator
  worker's CPU under ~80% or add workers/boxes.
